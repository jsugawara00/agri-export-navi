# 農産物輸出ナビ 実装指示書 v1.0（Claude Code向け）

- 対象: Claude Code（Cursor内）による実装
- 前提: 本書は企画書 v1.0 と対で機能する。判断に迷ったら企画書の設計原則に立ち返ること
- 進め方: 試作→人間が触って試す→リメイク。最初から完璧を狙わず、Phase 1の体験完成を最優先

---

## 0. 絶対原則（Toika共通・違反不可）

1. **AIは断定しない。最終確定は常に人間。** 検疫・制度情報には必ず「最終確認は植物防疫所等へ」の文言を添える
2. **判断基準はコードに埋め込まない。** 採点表・手順定義・乙仲リスト等はすべて `content/` 配下のmdファイルに置き、エンジンはそれをパースして動く。mdを書き換えるだけで判断が変わる構造を守る
3. **「輸出可能性◯%」という確率表現を実装しないこと。** ハードル指数は100点からの決定論的減点方式。内訳（減点項目・点数・対処案）を常に開示できるデータ構造にする
4. **スコア計算にLLMを使わない。** md採点表のパース＋加減算のみ（決定論的・再現可能・説明可能）。LLMの用途はメール下書き・契約書ひな形・文面生成に限定
5. **禁止品目は点数を出さない。**「輸出不可（相手国が輸入を禁止しています）」と可・不可を事実として区別表示
6. **全情報に鮮度メタデータ**（情報源・取得日・鮮度区分A/B/C）を持たせ、UIにバッジ表示
7. トレーサビリティ: 案件の状態変更（ステップ完了・入力値）は履歴として残す

## 1. 技術スタック

- TypeScript + Next.js（App Router）+ Vercel
- Firebase Auth（Googleプロバイダ）+ Firestore（Phase 2から導入。Phase 1は認証なしで動く）
- スタイリング: Tailwind。ダークトーン基調（#080d18系の濃紺＋ティール #8fd4b8 アクセント）
- 地球儀: **Canvas 2D の点描実装。Three.js・Mapbox等の3Dライブラリは使用禁止**（保守規模の制約）
- PDF生成: インボイス/PL/契約書出力用（ライブラリはPhase 3着手時に選定、まず@react-pdf/renderer を第一候補として検証）
- Claude API: サーバーサイド（Route Handler）経由でのみ呼ぶ。クライアント直呼び禁止

## 3. content/ mdファイルの共通フォーマット

すべてのmdは先頭にfrontmatterを持つ:

```yaml
---
freshness: B            # A | B | C
source: 植物防疫所 輸出入条件詳細情報
source_url: https://...
retrieved_at: 2026-07-17
reviewed_by: Jump       # 区分Bは人間確認済みであることを示す
---
```

採点mdの項目フォーマット（例 `criteria/institutional/apple_taiwan.md`）:

```md
## 減点項目
- id: orchard-registration
  points: -10
  label: 生産園地登録が必要
  remedy: 県の窓口経由で園地登録を申請する（ナビStep 3に対応）
  step_ref: apple_taiwan#step-3
- id: phytosanitary-cert
  points: -5
  label: 植物検疫証明書が必要
  remedy: 輸出前に植物防疫所の検査を受ける（ナビStep 5に対応）
  step_ref: apple_taiwan#step-5
```

- `step_ref` により減点項目とナビステップを必ず対応させる（ステップ完了→「ハードル解消済み」表示のため）
- 減点項目には**必ず remedy（対処案）を書く**。対処のない減点は原則作らない（作る場合は理由をmd内に明記）

## 4. 画面仕様（4画面）

### 4.1 トップ（検索）
- 地球儀: 緯度経度グリッドの点描。大陸判定で陸=ティール/海=暗グレー、**日本のみ別色**（例: アンバー系）。自動回転
- 入力: 品目プルダウン（米/りんご/ラ・フランス/その他の品目（対応のご相談））× 国プルダウン（台湾/香港/米国）。**自由入力なし**
- 検索実行: 地球儀が対象国方向へ回転→ズーム（scale 2.6, 900ms, cubic-bezier）→フェードアウト→結果へ
- コピー: メイン「調べる数ヶ月を、確かめる数分に。」／サブ「その一箱を、世界の食卓へ」
- フッター寄りにスコープ宣言＋相談導線（企画書4章の文言）
- 「その他の品目」選択時は問い合わせ案内を表示して終了（検索は走らせない）

### 4.2 結果（可否判定＋国情報）
- 国概要カード群（riseIn 500ms、60ms段差で順次出現）: 人口/通貨/言語/治安（外務省レベル連動）/検疫概要/EPA・関税
- 輸出ハードル指数: グレード（A〜E）＋点数＋一行評価（例「難易度B（78点）: 登録手続きは必要だが、実現ルートが確立されている組み合わせです」）
- 点数タップ→**内訳パネル**（減点項目・点数・対処案の一覧）
- 禁止組み合わせは点数非表示で「輸出不可」表示に分岐
- すべての情報カードに鮮度バッジ（区分＋取得日＋情報源）
- CTA: 「この内容でナビゲートを始める」（Phase 2以降は案件保存へ、Phase 1ではプレビューのみ）

### 4.3 ナビゲーション（案件画面・Phase 2）
- プログレスバー（あなたの現在地）＋準備完成度%（ステップ完了で上昇。ハードル指数とは別表示で混在させない）
- ステップリスト: 各ステップに「何のためにやるか」一行説明を常設
- 3層バッジ: 層1（システムで完結）/層2（下書きあり・実行はあなた）/層3（このアプリの範囲外）を視覚的に区別
- 官庁確認ステップ: 質問リスト表示→ユーザーが確認結果を入力→次へ進める（人間確認ゲート。未入力でスキップ不可）
- 再開時リキャップ: 「前回は◯◯まで完了。次は◯◯です」＋今日のTODO
- 減点項目に対応するステップ完了時: 内訳パネル側に「ハードル解消済み」を反映

### 4.4 書類生成（Phase 3）
- 英文輸出契約書ひな形: 変数箇所（品名・数量・単価・インコタームズ・支払条件等）をハイライト表示、フォームで差し替え→PDF出力。「一般的なひな形であり法的助言ではない。締結前にJETRO等へ相談」を出力物にも明記
- インボイス/パッキングリスト: フォーム入力→PDF出力
- 乙仲相談メール下書き: 案件データ（品目・国・港・数量・希望時期）から生成。**送信機能は実装しない**（コピーして本人が送る）
- 港選定: 最寄り港＋東京港の2択方式→ `forwarders/{port}.md` から乙仲リスト表示

## 5. 採点エンジン仕様（lib/score）

```
input: { item, country } 
処理: institutional/{item}_{country}.md + geopolitical/{country}.md + logistics/{route}.md を読込
     → 禁止フラグがあれば { status: "prohibited" } を返して終了
     → 100 + Σ(points) を計算、グレード判定（A:90+ / B:75+ / C:60+ / D:40+ / E:40未満）
output: { status, score, grade, summary, breakdown: [{id,points,label,remedy,step_ref,resolved}] }
```

- 純関数として実装し、ユニットテスト必須（禁止分岐/減点合算/グレード境界/step_ref整合）
- 外務省危険情報レベル（区分A）はPhase 1ではmd内の静的値でよい。取得自動化はPhase 4

## 6. 実装順序

1. **Phase 1**: content/雛形（3品目×3カ国分）→採点エンジン＋テスト→トップ（地球儀・演出）→結果画面（内訳パネル・鮮度バッジ）
2. **Phase 2**: Firebase Auth/Firestore→案件保存→ナビ画面（ステップ/TODO/リキャップ/完成度）
3. **Phase 3**: PDF生成3種＋乙仲リスト＋メール下書き＋銀行チェックリスト
4. **Phase 4**: 区分B巡回・差分検知（検知→運用者確認キュー→md更新提案。自動書換は禁止）

各Phase完了時にビルド・テスト・簡易動作確認を行い、コミットを分けること。

## 7. Firestoreデータモデル（Phase 2）

```
users/{uid}
projects/{projectId}: { uid, item, country, buyerMemo, createdAt, updatedAt,
                        hurdle: {score, grade, snapshotAt},   # 判定時点のスナップショット
                        progress: { completedSteps: [], completionPct },
                        inputs: { ... },                       # 書類用入力値
                        history: [ {at, action, stepId} ] }    # トレーサビリティ
```

- ハードル指数は**判定時点のスナップショット**として案件に保存（後でmdが更新されたら「基準が更新されています。再判定しますか？」を表示。無断で数字を差し替えない）

## 8. 禁止事項

- Three.js等3Dライブラリの導入 / 自由入力フィールド / 確率表現 / スコアのLLM算出
- メール自動送信・官庁への自動問い合わせ（層2の実行は必ず人間）
- content/ の内容をTS内に定数として複製すること（単一情報源はmd）
- 鮮度メタデータのない情報表示
- 「最終確認は◯◯へ」文言の省略

## 9. テスト・品質

- 採点エンジン: ユニットテスト必須（上記4観点）
- content/ ローダ: frontmatter欠落・step_ref不整合を検出するバリデーションテスト
- E2E（Phase 2以降）: 検索→判定→保存→再開の主要動線
- コミットメッセージは日本語可・変更意図を書く

## 10. 開発中の判断ルール

- 仕様の曖昧さに直面したら: 企画書の該当章を参照→それでも不明なら実装を止めて質問を列挙（勝手に仕様を発明しない）
- 見た目の演出とロジックの正確さが衝突したら: 正確さを取る（演出は後から盛れる）
- スコープが膨らみそうになったら: 「山形3品目×3カ国・md追加で拡張」の原則に戻る
