## 1. プロジェクト概要

| 項目 | 内容 |
|---|---|
| 仮称 | 農産物輸出ナビ（Toika Export Navigator） |
| ブランド | Toika（問うて、確かめる） |
| メインコピー案 | 「調べる数ヶ月を、確かめる数分に。」 |
| サブコピー案 | 「その一箱を、世界の食卓へ」 |
| 形態 | Webアプリ（デモ／ポートフォリオ→自治体・企業向け再設計の営業導線） |

### ミッション
利益が出るか出ないか以前に、「農家や個人事業だけでは面倒でハードルが高くて無理」という輸出の壁を、そっと、しかも確実に後押しする。入口はカッコよく引き込み、内容を見ると「え、これできるんじゃないの？」と思わせ、最後は手順通りに進めばゴールに辿り着ける。

### このアプリの価値の定義
**リスクをゼロにすることではなく、「わからない」に費やす時間を圧縮すること。**
早期に「無理」と分かることも価値（無駄な数ヶ月を防ぐ）。

## 2. 背景と課題

- 農産物輸出に必要な情報は国が整備済み（植物防疫所の輸出入条件詳細情報・早見表、JETRO輸出支援ポータル等）だが、省庁・窓口ごとにバラバラで、素人が読んでも「結局うちの品目はこの国に送れるのか」が分からない。
- 関係先が多岐にわたる: 税関（通関）、農水省・植物防疫所（検疫）、厚労省（食品衛生）、外務省（渡航・情勢）、JETRO、経産省（輸出貿易管理令・稀に該当）、環境省（CITES・品目拡大時に該当可能性）、都道府県（個別申請がある場合）。
- データは揃っているのに「ナビゲーションに変換する層」が存在しない。ここが本アプリの空白地帯。
- 想定される典型場面: **商談会でバイヤーと知り合った個人社長・農家が、その先の進め方が分からない。**

---

## 3. ターゲットユーザーとユーザーストーリー

### プライマリペルソナ
「具体的な引き合いがあるのに、その先の進め方が分からない人」
- 小規模農家・個人社長・農業法人（1〜10名規模）
- 商談会・マッチングイベントでバイヤーと接点ができた直後
- IT習熟度は高くない。本業（農作業）の合間の細切れ時間で準備を進める

### 代表ユーザーストーリー
> 商談会で台湾のバイヤーに「おたくのりんごが欲しい」と言われた。帰りの新幹線でスマホからこのアプリを開き、品目と国を選んで検索。数分で「輸出可能・難易度B・必要なのは生産園地登録と植物検疫証明書」と分かる。その日から案件として保存し、週末ごとにTODOを1つずつ潰して、数週間後に乙仲へ相談メールを送るところまで辿り着く。

### セカンダリターゲット（営業対象）
- 自治体（農産物輸出振興の担当部署）、農協、地域商社
- 「貴自治体の特産品を盛り込んだ再設計」の相談導線を最初から組み込む

## 4. スコープ（MVP）

### 対象品目 × 対象国（デモ想定）
| | 台湾 | 香港 | 米国 |
|---|---|---|---|
| 米 | ○ | ○ | ○ |
| りんご | ○（生産園地登録あり＝ナビしがいのある要件） | ○（検疫ほぼ不要＝成功体験ルート） | ○ |
| ラ・フランス | ○ | ○ | ○ |

- 難易度が松竹梅で揃う3カ国を選定（香港=易、台湾=中、米国=難）
- 入力は**選択式（プルダウン）に限定**。自由入力は採用しない
  - 理由: 「調べた結果ダメでした」体験を作らない。答えられる範囲を入力UIで宣言する
  - 品目プルダウンに「その他の品目（対応のご相談）」を1つ用意し、選択時は問い合わせ案内へ
- トップページにスコープ宣言を常設:
  > 本アプリは山形県産の米・りんご・ラ・フランスを対象としたデモンストレーションです。貴自治体・貴社の特産品を盛り込んだ再設計のご相談を承ります。

### 非スコープ（層3: システムは関与しない。UI上で明示的に宣言）
- バイヤーとの価格交渉・契約締結そのもの
- 送金の実行
- 検疫の実地検査、生産園地登録の実地対応
- 輸出後のリスク・採算性の保証
- 法的助言（契約書はあくまで一般的ひな形。JETRO等の専門相談へ橋渡し）

## 5. 全体フロー（7フェーズ）

1. **検索・可否判定** — 地球儀→品目×国選択→ズーム演出→国概要＋検疫概要＋輸出ハードル指数
2. **詳細要件の展開** — 検疫条件・施設/園地登録の要否・食品衛生・EPA税率・必要書類一覧（すべて鮮度バッジ付き）
3. **シミュレーション開始** — 案件として保存し、ステップ管理画面へ。「あなたの現在地」プログレスバー
4. **官庁確認フェーズ** — 防疫所等への確認事項リスト＋質問文面を生成。確認結果をユーザーが入力して次へ（人間確認ゲート）
5. **書類生成フェーズ** — 英文輸出契約書ひな形（変数ハイライト方式）、インボイス、パッキングリスト（フォーム入力→PDF出力）
6. **物流手配フェーズ** — 港選定（最寄り＋東京港の2択方式。山形なら酒田・仙台塩釜・新潟・東京）→乙仲リスト提示→相談メール下書き生成
7. **最終チェック** — 銀行相談チェックリスト（L/C・前払い・掛け売りの決済方式とリスク解説を含む）、JETRO相談案内、残タスク（契約・送金）の明示

## 6. 中核設計①: 2軸の数字（混ぜない）

| 数字 | 意味 | 性質 |
|---|---|---|
| **輸出ハードル指数**（難易度グレードA〜E＋点数） | 品目×国の組み合わせ自体の難易度 | 検索した瞬間に決まる静的判定 |
| **準備完成度（%）** | ユーザーの準備がどこまで整ったか | ステップを進めるごとに上がる進捗 |

- 「可能性◯%」という**確率表現は使わない**（統計的裏付けが存在しない数字を演出しない）
- 禁止品目は点数を出さず「輸出不可（相手国が輸入を禁止しています）」と明確に区別する
- 可・不可は事実、点数は難易度。この2つを混ぜない

## 7. 中核設計②: ハードル指数の減点方式（説明可能スコア）

100点満点から、**実在する具体的な要因**で減点する。数字をタップすると内訳が開き、「引かれた22点はこの4項目です。それぞれこう対処できます」と表示。減点には必ず対処案を添える（保険・L/C・園地登録など）。減点項目はそのままナビゲーションのステップに直結し、完了すれば「ハードル解消済み」表示になる。

### 減点3軸
| 軸 | 例 | 更新区分 |
|---|---|---|
| 制度要因 | 検疫条件、園地登録、EPA有無、食品衛生要件 | 区分B（変更検知型） |
| 国情勢要因 | 戦争・紛争、政情不安、経済制裁、為替不安定（外務省危険情報レベル連動。レベル3〜4は大幅減点または輸出非推奨表示） | 区分A寄り（都度取得） |
| 物流経路要因 | 海賊多発海域の経由（ソマリア沖・アデン湾、マラッカ海峡、ギニア湾等）→減点＋「戦争・S.R.C.C.等の保険特約推奨」を提示 | 区分C＋定期見直し |

### 判断基準の外置き（Toika共通原則）
採点表（何を何点減点するか）はコードに埋め込まず、**判断基準mdファイル**として保持。エンジンは触らずmdの更新だけで判断が変わる。問われたら「判断根拠はすべて基準ファイルに分類・明文化されており、AIはそれを参照して採点している」と説明できる（RAG的分類の明示）。

## 8. 中核設計③: 情報鮮度の3区分（全情報に鮮度バッジ）

| 区分 | 内容 | 例 | 更新方法 |
|---|---|---|---|
| A: 自動取得 | 検索時に都度取得 | 為替レート、外務省危険情報レベル、国の基本情報 | API・公開データ |
| B: 変更検知型 | 定期巡回＋差分検知＋**人間確認後に**md更新 | 植物検疫条件、EPA税率、食品衛生要件 | 巡回→AI差分検知→運用者確認→md反映 |
| C: 静的md | 手動整備・更新日明記 | 手順定義、自治体×特産品、乙仲リスト、銀行チェックリスト | 手動 |

- UI表記例: 「この情報は◯月◯日時点の植物防疫所公表資料に基づく」
- 植物防疫所自身が「頻繁な改正に完全対応できない」と明記しているため、**「最終確認は植物防疫所へ」の一文を必ず添える**（三層防御思想のデータ版）

## 9. 中核設計④: ナビゲートの3層（誠実さの線引き）

| 層 | 内容 | 例 |
|---|---|---|
| 層1: システムが完結 | 判定・生成・管理 | 可否判定、必要書類リスト、チェックリスト進捗、インボイス/PL生成、契約書ひな形生成、乙仲メール下書き |
| 層2: 文面と段取りは作るが実行は人 | 下書き＋チェックリスト提供 | 防疫所への確認電話（質問リスト付き）、乙仲へのメール送信（本人が補正して送信）、JETRO相談予約、銀行外為窓口への事前相談 |
| 層3: システムは関与しない（明示宣言） | 人間の領域 | 価格交渉・契約締結、送金実行、検疫実地検査、園地登録の実地対応 |

## 10. 保存・再開・伴走機能

- 案件（品目×国×バイヤー）単位でプロジェクト保存
- 再開時リキャップ: 「前回は園地登録の確認まで完了。次は防疫所への確認電話です」
- 今日のTODOリスト: 層1の準備済み項目と層2の人間タスクが混在して並ぶ
- 各ステップに「これは何のためにやるか」を一行で常設（数週間ぶりでも迷子にならない）
- 準備完成度%がステップ完了ごとに上昇

## 11. UI・デザイン方針

- コンセプト: グローバルで知的でカッコいい。入口で引き込み、作業画面は淡々と信頼感重視（入口と中身で温度を変える）
- トップ: 点描ドットスフィアの地球儀
  - **大陸準拠の陸地ドット（陸=明るいティール、海=暗いグレー）、日本のみ別色**
  - 検索→対象地域へ回転・ズーム→フェードアウト→情報カードが順に浮かび上がる（riseIn、60ms段差）
  - Three.js等の重い3D実装は使わない。軽量Canvas点描＋固定演出（1人で保守できる規模を維持）
- 配色: 濃紺基調＋ティールアクセント。発光・ぼかしに頼らず、密度と動きのタイミングで魅せる
- キャッチコピー: メイン「調べる数ヶ月を、確かめる数分に。」＋サブ「その一箱を、世界の食卓へ」
  - コピーが性能要件を規定する: **検索→ハードル指数表示まで数分以内（体感即時）**
- スコープ宣言と相談導線をフッター寄りに常設

## 12. 技術構成

| 要素 | 選定 | 理由 |
|---|---|---|
| フロント/フレームワーク | TypeScript + Next.js (App Router) | 検問所と同一。既存資産・習熟 |
| 認証・DB | Firebase Auth (Google) + Firestore | そっと。の構成を流用（案件保存・再開に必須） |
| AI | Claude API | メール文面生成・契約書ひな形生成・確認チャット。**スコア計算はAI推論ではなく決定論的計算**（md採点表のパース＋加減算） |
| 判断基準 | mdファイル群（リポジトリ外の私有分を含む運用は要検討） | Toika共通原則 |
| PDF生成 | インボイス/PL/契約書の出力 | フォーム入力→出力方式 |
| ホスティング | Vercel | 検問所と同一 |

### データ（md）構造方針
- `criteria/institutional/{品目}_{国}.md` — 制度要因（品目×国マトリクス。自治体・品目md群と紐づけ）
- `criteria/geopolitical/{国}.md` — 国情勢要因（品目非依存）
- `criteria/logistics/{航路}.md` — 物流経路要因（品目非依存）
- `data/municipalities/{自治体}.md` — 自治体×特産品パターン（将来の横展開単位。企業版は自治体を企業に置換するだけ）
- `data/forwarders/{港}.md` — 乙仲リスト（税関公表の通関業者一覧・通関業連合会検索システム等の公開情報から整備）
- `data/procedures/{品目}_{国}.md` — ステップ定義（減点項目とステップの対応を明記）
- 更新頻度・責任範囲の異なる情報を同一ファイルに混ぜない（そっと。のPII分離と同じ発想）

## 14. 成功基準（デモ段階）

- 検索からハードル指数表示まで体感即時（キャッチコピーと矛盾しない）
- ハードル指数の全減点項目が内訳として開示され、対処案が添えられている
- 全情報に鮮度バッジと情報源・取得日が表示される
- 中断→再開時にリキャップとTODOが機能する
- 自治体・農協担当者が触ったとき「うちの品目でも作れるか」という問い合わせ導線が機能する

## 15. 将来展望

- 自治体単位のmd追加による横展開（「自治体の売りたいものパターン」方式。企業版は主語を置換するだけ）
- 「バラバラな公的情報を現場で使える形に翻訳・構造化する」型として、他の規制産業への応用可能性
- **開発者自身によるドッグフーディング輸出**（ラ・フランスの季節に、本アプリの手順のみで山形から一箱送る）→最強の検証＋営業実績
- 検問所・そっと。とのブランド一貫性: 「AIは断定しない」「人間確認ゲート」「判断基準の外置き」「トレーサビリティ」

## 16. リスクと対応

| リスク | 対応 |
|---|---|
| 相手国規則の頻繁な改正に追随できない | 鮮度バッジ＋「最終確認は防疫所へ」の常設。区分Bの変更検知運用。断定しない文言設計 |
| スコアの数字が独り歩きする | 確率表現の禁止。減点方式＋内訳全開示＋グレード表記。禁止品目は点数を出さない |
| 対応外品目への期待 | 選択式入力＋スコープ宣言＋相談導線で入力時点で期待値制御 |
| 法的責任（契約書・助言） | 一般的ひな形である旨の明示、JETRO等専門窓口への橋渡し、層3の明示宣言 |
| 開発規模の発散 | 山形3品目×3カ国に固定。md追加による拡張はリメイク以降 |
