# 巡回→修正案PR の手順（手元実行）

外務省（www.mofa.go.jp）はデータセンターIPを403で拒否するため、
mofa系3ページの巡回は必ず運用者の手元PC（家庭のIP）で行う。

## 手順

1. `git checkout main && git pull --ff-only` で最新化
2. `npm run patrol` を実行
3. 結果で分岐:
   - 変化なし → その旨を報告して終了（コミット不要。patrol-report.md の
     日付更新だけが残った場合は main に直接コミットしてよい）
   - 変化あり → 手順4へ
4. 変化があった各情報源について:
   - `git diff -- ops/snapshots/` で旧→新を読み、日本語で差分を要約
   - キューmdの targets にある content/ のmdを読み、修正案を作成
     （retrieved_at も更新）。**禁止品目の追加・解除、減点数値の変更など
     重大な変更は修正案を出さず「要相談」として差分要約のみ**
   - 国概要md（countries/）への追記は、UIに描画される既存の節
     （治安・検疫概要・EPA・関税）の中に書くこと。新しい節名は
     loader.ts の抽出対象外で画面に表示されない（2026-07-18の教訓）
   - 影響なしと判断した場合はキューmdに理由を追記して resolved にする
   - **データの基準日（frequency_as_of ／ 本文の「◯◯時点」「◯年◯月◯日現在」）は、
     巡回日ではなく「出典ページ自身の更新日／基準日」を入れること**（例: ページに
     「2026年1月1日現在」とあればその日付、航路表画像の日付があればそれ）。出典が
     更新日を明示しない場合は、その旨を出典表記に添える。retrieved_at は従来どおり
     「こちらが確認した日」でよい（＝人間確認日。データの新しさとは別物）
4.5. **到達できない情報源があった場合**（レポートに `UNREACHABLE`／`✗ ERROR`）:
   - サイトの移転・閉鎖の可能性。同等の公的／公表情報の代替先を探す
     （港湾なら港湾管理者・港湾協会・地方整備局の公表ページ等）
   - 対象mdの `source_url` を代替先に更新する案を作り、PR本文に
     「到達不能→代替提案」節として旧URL・新URL・根拠を明記する
   - 代替が見つからない場合は「要相談」として状況のみPRに残す
   - この状況はクラウド見張り番が patrol-report.md をgrepしてメール通知する
5. `npm test` 全件PASSを確認
6. ブランチ `patrol/YYYYMMDD` を作成し、ops/ と content/ の変更をコミット、
   `gh pr create` で main に向けたPRを起票。タイトル
   「[patrol] YYYY-MM-DD 情報源の変更検知（N件）」
7. PR本文（すべて日本語）:
   - 冒頭に2択「■承認→Merge（自動デプロイ）」「■相談→マージせず手元で確認」
   - 変更ごとに【旧→新】要約（情報源URL付き）、修正案と根拠、要相談の理由
   - 末尾に「※最終確認は植物防疫所等の公的機関へ。修正案はAIの下書き」
8. 運用者にPRのURLを渡し、Merge（承認）または相談を促す
